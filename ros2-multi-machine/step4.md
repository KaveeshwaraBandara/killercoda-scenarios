# The laptop side: your Stage 2 code, unchanged

Here's the moment the whole course was built for.

In Stage 2 you wrote `square_driver.py` to drive a **simulated** robot in Gazebo. It publishes `Twist` messages to `/cmd_vel`. It knows *nothing* about Gazebo, and nothing about hardware — it just publishes to a topic.

So it will drive the motor driver on the robot machine **without a single edit**. Not a port. Not an adaptation. The same file.

```bash
cat > /root/work/square_driver.py <<'EOF'
import time
import rclpy
from rclpy.node import Node
from rclpy.signals import SignalHandlerOptions
from geometry_msgs.msg import Twist


class SquareDriver(Node):
    """Drives: forward 1m, turn 90 degrees, repeat. (Stage 2, unmodified.)"""

    EDGE_TIME = 4.0     # 4 s at 0.25 m/s  -> 1 m
    TURN_TIME = 3.1416  # pi s at 0.5 rad/s -> 90 degrees

    def __init__(self):
        super().__init__('square_driver')
        self.publisher = self.create_publisher(Twist, 'cmd_vel', 10)
        self.timer = self.create_timer(0.1, self.step)   # 10 Hz control loop
        self.state = 'forward'
        self.state_time = 0.0

    def step(self):
        self.state_time += 0.1
        cmd = Twist()

        if self.state == 'forward':
            cmd.linear.x = 0.25
            if self.state_time >= self.EDGE_TIME:
                self.switch('turn')
        else:
            cmd.angular.z = 0.5
            if self.state_time >= self.TURN_TIME:
                self.switch('forward')

        self.publisher.publish(cmd)

    def switch(self, new_state):
        self.get_logger().info(f'{self.state} done -> {new_state}')
        self.state = new_state
        self.state_time = 0.0

    def stop_robot(self):
        self.publisher.publish(Twist())   # an all-zero Twist means "stop"


def main():
    # Ctrl+C is OURS to handle — see the note at the end of this step.
    rclpy.init(signal_handler_options=SignalHandlerOptions.NO)
    node = SquareDriver()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        node.get_logger().info('Ctrl+C — stopping the robot')
        node.stop_robot()
        time.sleep(0.2)
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
EOF
echo "square_driver.py written — identical to the Stage 2 version"
```{{exec}}

Run it on the **laptop**, and watch the wheels on the **robot**:

```bash
docker exec -d laptop bash -lc "python3 /work/square_driver.py > /tmp/square.log 2>&1"
sleep 12 && docker exec robot bash -lc "tail -n 6 /tmp/motor.log"
```{{exec}}

Wheel speeds, computed on the robot, from commands generated by simulator code running on another machine. Wait a few seconds and look again — the robot has reached a corner:

```bash
sleep 8 && docker exec robot bash -lc "tail -n 4 /tmp/motor.log" && echo "--- what the driver thinks it's doing: ---" && docker exec laptop bash -lc "tail -n 3 /tmp/square.log"
```{{exec}}

Equal-and-opposite wheel speeds — it's turning the corner of its square. **Same code, different robot.** That's the payoff of publishing to an interface instead of to a machine.

## The stop that matters

Kill the driver, then look at the very last command the robot's wheels received:

```bash
docker exec laptop bash -lc "pkill -INT -f square_driver.py"
sleep 3 && docker exec robot bash -lc "tail -n 1 /tmp/motor.log"
```{{exec}}

`wheels(L= +0.00 R= +0.00)` — **the robot stopped.** The driver published a zero-velocity `Twist` on its way out.

Without that, the motor driver's last instruction would be *"keep driving"* — and **a robot obeys its last command forever.** Your program is gone; the robot is still going. In simulation that's a bug; on hardware it's a robot leaving the building.

And notice the line that makes it actually work:

```python
rclpy.init(signal_handler_options=SignalHandlerOptions.NO)
```

The *obvious* version of this code — plain `rclpy.init()`, publish zero in the `except` block — **silently fails**. rclpy's default Ctrl+C handler destroys the ROS context before your shutdown code runs, so the stop message dies with `publisher's context is invalid` and never reaches the robot. Taking over signal handling keeps the context alive long enough to send it.

!!! tip "The professional backstop: a command watchdog"
    Real motor drivers don't *trust* the sender to say goodbye. They run a **timeout**: no `/cmd_vel` for ~0.5 s → stop the motors. Then a crashed driver, a dead laptop battery, or WiFi dropping mid-corner all fail *safe*. If your robot's driver has a `cmd_vel_timeout` parameter, that's what it's for — leave it on.
